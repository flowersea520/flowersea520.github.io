git工作区，暂存区，版本库概念

包含了 .git文件的目录就是本地仓库目录，也叫工作区，

git 远程仓库操作命令

git remote：查看远程仓库：origin （默认远程仓库简称）– 由于是克隆下来的，所以才能看到远程仓库名

git remote -v查看远程仓库对应的网路地址：（每一个远程仓库都有对应的网络地址）

如果不是远程仓库克隆下来的，则使用git remote不会显示对应的远程仓库

git remote add <简称> URL将本地仓库添加到远程仓库：一般这里的简称用origin

将本地麦吉项目初始化（git init）为本地仓库后，添加远程仓库命令：添加成功后使用git remote查看远程仓库

git push 【remote-name–就是origin】【branch-name– 分支名，master或者其他分支】将本地仓库内容推送到远程仓库

其中remote-name可以使用：git remote命令去查看，一般都是origin

代码推送到远程仓库演示：

先查看本地仓库文件状态，是否有未添加到暂存区的文件git add或者为提交到(git commit)本地仓库的文件

提交到本地仓库，

git status显示本地仓库所有的文件都已经提交

git push origin master: 推送到仓库的主分支上面去

如果创建了一个文件，或者新更新（编辑）了一个文件，就要重新添加到暂存区，然后提交commit到本地仓库

如果线上代码（远程仓库）和线下（本地仓库）代码不一致就会push推送报错（之所以出现这个原因，是因为我在线上生成、编辑了README.md文件，而本地代码文件中不包含它，所以线上线下就对不上了。）

git pull –rebase origin master

使用git pull –rebase origin master（分支名），即可将线上、线下代码进行合并

然后再进行push：git push origin master

git pull origin 【branch-name】远程仓库拉取

如果远程仓库和本地仓库代码一直，则不需要拉取

标签可以回到标记时的状态（某个分支特定时间点的状态）：

  • 新建名字叫v0.1的标签：git tag [name]
  • 将标签推送到远程仓库中去
  • 将v0.1的标签推送到远程仓库：

查看github的表情，发现推送push过来了（这个标签记录的是main分支的内容状态）只有三个文件的状态（v0.1记录的）

  • 现在在本地仓库创建一个新的文件（0.2tag文件），然后commit到本地仓库，再次打上标签v0.2（这个v0.2的标签记录的是，有0.2tag文件的状态）

  • 提交完之后，再打标签，才能生效

  • 将v0.2标签推送到远程仓库

  • 发现多了一个标签

检出标签：将某个标签的文件代码弄出来，到分支上

git checkout -b [branch – 分支名(新建的)] [name(标签名)]

  • 使用该命令后，会自动创建对应分支(b0.1)，然后将0.1标签状态的代码弄到0.1分支上去了（当时0.1状态没有v0.2tag文件）
  • 检出0.2 标签成功（因为多了v0.2tag文件）

分支和标签的区别

  • 标签相当于是一个照片，是一个静态的概念，一旦打好了标签，他就像图片显示一样定格住了，不能改变（一般标签使用：开发好了软件，开发了一个新版本后就打了标签，每个版本一个标签）

  • 而分支是动态的概念：他能改变（比如java代码里面有10个类，明天可以弄20个类，上传到远程仓库）
